*** Findings from debug_rlm.py (cached run)

Running =debugging/debug_rlm.py= with cached LLM responses (all =generate_action=
calls complete in 0.0s) reveals a second hang source:

- Iterations 1-5 complete instantly (code interpreter runs fast)
- *Iteration 6 hangs in =repl.execute()=*: the model generates a =while idx < len(context)=
  loop doing character-by-character brace matching over 144k chars. The Pyodide/WASM
  code interpreter cannot handle this at reasonable speed.

The 4B model fails to parse the JSON in iterations 1-4 (the context is comma-separated
JSON objects without a wrapping =[]= array, so =json.loads()= fails with "Extra data").
By iteration 5-6, it resorts to manual character-by-character parsing, which is
computationally infeasible in the sandboxed interpreter.

*** Confirmed causes

Two independent hang sources:

1. *Code interpreter blocks on expensive loops* (confirmed): the model generates O(n)
   character iteration code over 144k chars, which the WASM sandbox cannot execute
   in reasonable time. This would happen even with a fast LLM.

2. *LLM timeout not enforced* (from Phoenix traces): 3 errored spans show
   =litellm.Timeout= firing at 434-449s instead of the configured 60s. For example 71,
   the timeout may never fire at all. This is a separate issue from #1 but compounds it.

50 errors clustered in 41 traces (some examples retried the same broken
~llm_query~ call up to 10x).  *Root cause is in the model's generated
code, not in our infrastructure*: the model called
~llm_query(some_dict)~ instead of ~llm_query(some_string)~.  The dict

~llm_query~ -> ~target_lm(prompt=some_dict)~ -> litellm packaged it as

OpenAI-compatible endpoint rejected it because ~content~ must be a
string or a list of content parts.

declared as ~def llm_query(prompt: str)~ but Python type hints don't
enforce -- the dict slips through.  A one-line guard in
~rlm._make_llm_tools.llm_query~ (raise ~TypeError~ on non-str input
with a self-explanatory message) would surface the problem to the
model immediately as a REPL observation, instead of bouncing through
litellm and llama.cpp.  Filed-worthy follow-up; no task yet.
