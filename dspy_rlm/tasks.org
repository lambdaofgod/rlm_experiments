#+title: DSPy SFT Data Collection - Tasks

* Overview

Collect SFT training data from a DSPy program (teacher LM), filter by
metric, and write JSONL in DSPy's ChatAdapter format -- ready for
finetuning with Unsloth/TRL. Based on the ~SFTDataCollector~ pattern
from [[file:dspy_sft_data_collection.org][dspy_sft_data_collection.org]]. The DSPy module type
(ChainOfThought, RLM, etc.) is configurable via YAML.

** DONE Abstract tracing backend -- decouple from Arize Phoenix

   Phoenix is currently hardcoded in two roles:

   1. *Trace writer* (~collect_sft_data.py~): uses ~phoenix.otel.register~
      + ~DSPyInstrumentor~ to send OTEL spans.  This is thin -- Phoenix
      is just one OTEL collector endpoint, any OTEL-compatible backend
      (Jaeger, Langfuse, Datadog, stdout) could replace it.

   2. *Trace reader* (~evaluate.py~, ~export_traces.py~): uses
      ~phoenix.client.Client~ to fetch spans as a pandas DataFrame.
      This is deeply coupled -- column names (~attributes.input.value~,
      ~status_code~, ~context.trace_id~, ~parent_id~), the
      ~get_spans_dataframe~ API, query parameters (~root_spans_only~,
      ~project_name~), and the DataFrame schema are all Phoenix-specific.

*** Goal

    Be able to swap Phoenix for another OTEL backend (Langfuse, Jaeger,
    local SQLite, flat JSONL files) without touching evaluation or
    export logic.

*** Approach

    Split into two interfaces:

**** 1. Trace writer abstraction (~tracing.py~)

     Replace ~setup_tracing()~ with a configurable factory.

     #+BEGIN_SRC python
     def setup_tracing(backend, endpoint, project_name=None):
         if backend == "phoenix":
             from phoenix.otel import register
             tracer_provider = register(endpoint=endpoint, project_name=project_name)
         elif backend == "otel":
             # Generic OTEL exporter (works with Jaeger, any OTLP collector)
             from opentelemetry.sdk.trace import TracerProvider
             from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
             from opentelemetry.sdk.trace.export import BatchSpanProcessor
             tracer_provider = TracerProvider()
             tracer_provider.add_span_processor(
                 BatchSpanProcessor(OTLPSpanExporter(endpoint=endpoint))
             )
         elif backend == "none":
             return
         DSPyInstrumentor().instrument(tracer_provider=tracer_provider)
     #+END_SRC

     Config change: add ~traces_backend~ field.

     #+BEGIN_EXAMPLE
     traces_backend: "phoenix"   # or "otel", "none"
     traces_endpoint: "http://localhost:6006/v1/traces"
     traces_project: "my-project"
     #+END_EXAMPLE

**** 2. Trace reader abstraction (~tracing_backend.py~)

     Define an abstract ~TracingBackend~ base class that ~evaluate.py~ and
     ~export_traces.py~ use instead of calling ~phoenix.client~
     directly.

     #+BEGIN_SRC python
     from abc import ABC, abstractmethod

     class TracingBackend(ABC):
         @abstractmethod
         def get_root_spans(self, project_name: str, limit: int) -> pd.DataFrame: ...

         @abstractmethod
         def get_all_spans(self, project_name: str, limit: int) -> pd.DataFrame: ...

     class PhoenixBackend(TracingBackend): ...
     class JsonlBackend(TracingBackend): ...  # future
     #+END_SRC

     The returned DataFrames must have a canonical schema:

     | Column                   | Type     | Description                           |
     |--------------------------+----------+---------------------------------------|
     | span_id                  | str      | Unique span identifier                |
     | trace_id                 | str      | Groups spans into a trace             |
     | parent_id                | str/None | Parent span ID                        |
     | name                     | str      | Span name (e.g. "RLM.forward")        |
     | status_code              | str      | "OK" or "ERROR"                       |
     | status_message           | str/None | Error message if status=ERROR         |
     | start_time               | datetime | Span start                            |
     | end_time                 | datetime | Span end                              |
     | input_value              | str      | JSON string of input args             |
     | output_value             | str      | JSON string of output                 |

     Implementations:
     - ~PhoenixBackend~: wraps current ~phoenix.client~ code, maps
       Phoenix column names to canonical schema
     - ~JsonlBackend~ (future): reads spans from a local JSONL file
       (useful for offline eval without a running Phoenix instance)

     The column renaming happens once inside each ~TracingBackend~
     implementation.  All downstream code (~evaluate.py~,
     ~export_traces.py~) uses only the canonical column names.

*** Trace reader implementation (~tracing_backend.py~)

    Implement ~MlflowBackend(TracingBackend)~:

    #+BEGIN_SRC python
    class MlflowBackend(TracingBackend):
        def __init__(self, endpoint: str | None = None):
            import mlflow
            if endpoint:
                mlflow.set_tracking_uri(endpoint)
            self._client = mlflow.client.MlflowClient()

        def get_root_spans(self, project_name, limit=10000):
            traces = self._client.search_traces(
                experiment_ids=[self._get_experiment_id(project_name)],
                max_results=limit,
            )
            rows = []
            for trace in traces:
                root = trace.data.spans[0]  # root span
                rows.append(self._span_to_row(root, trace.info.request_id))
            return self._to_dataframe(rows)

        def get_all_spans(self, project_name, limit=10000):
            traces = self._client.search_traces(
                experiment_ids=[self._get_experiment_id(project_name)],
                max_results=limit,
            )
            rows = []
            for trace in traces:
                for span in trace.data.spans:
                    rows.append(self._span_to_row(span, trace.info.request_id))
            return self._to_dataframe(rows)
    #+END_SRC

    Key mapping details:
    - ~trace.info.request_id~ -> ~trace_id~
    - ~span.span_id~ -> ~span_id~
    - ~span.parent_id~ -> ~parent_id~ (None for root)
    - ~span.name~ -> ~name~
    - ~span.status.status_code.name~ -> ~status_code~ ("OK" / "ERROR")
    - ~span.status.description~ -> ~status_message~
    - ~span.start_time_ns~ / ~span.end_time_ns~ -> timestamps (convert from ns)
    - ~json.dumps(span.inputs)~ -> ~input_value~
    - ~json.dumps(span.outputs)~ -> ~output_value~

    *Important*: MLflow's ~span.inputs~ and ~span.outputs~ are dicts,
    not JSON strings.  Must ~json.dumps~ them to match the canonical

*** Background

    DSPy supports structured output through ~JSONAdapter~, which uses
    LiteLLM's ~response_format~ parameter.  The level of enforcement
    depends on the model provider prefix:

    | Provider prefix | ~response_format~ sent   | Server-side enforcement |
    |-----------------+--------------------------+-------------------------|
    | ~openai/~       | ~{"type": "json_object"}~ | JSON only (no schema)   |
    | ~lm_studio/~    | Pydantic JSON schema     | Full schema             |

    For local OpenAI-compatible servers (llamacpp, vLLM, SGLang), the
    ~lm_studio/~ prefix is needed for full structured output -- it is in
    LiteLLM's hardcoded ~PROVIDERS_GLOBALLY_SUPPORT_RESPONSE_SCHEMA~
    list, so ~supports_response_schema~ returns True.

    With ~openai/~ prefix, ~JSONAdapter~ falls back to basic JSON mode:
    the schema is included in the prompt text but not enforced server-side.
    Both work for nested Pydantic models, but ~lm_studio/~ gives grammar-
    constrained generation.

** DONE Skip already-traced examples in ~collect_sft_data.py~

   When resuming a long collection run (e.g. after a crash, server restart,
   or to extend a partial run), every example is re-processed from scratch.
   RLM calls are expensive (multiple LM iterations + code interpreter per

*** Histogram format

    Fixed-width layout so the right-side ~[pct% (n)]~ annotation
    lines up across rows.  Bar area is 40 chars wide (scales to
    100%, so each ~#~ = 2.5%); bar is left-aligned and padded with
    spaces to fill the area; two spaces of gap; then the right-side
    annotation ~[%3d%% (%d)]~ -- 3-digit zero-padded-by-space percent
    so ~0~, ~10~, and ~100~ all end at the same column before ~%~.

    Layout per line:
    ~"%.1f - %.1f %-40s  [%3d%% (%d)]"~

    Where the first ~%-40s~ holds the bar (e.g. ~"####"~) padded on
    the right to 40 chars.  Worked example:

    #+BEGIN_EXAMPLE
    *** accuracy histogram (n=95)
    #+begin_example
    0.0 - 0.1 ################################          [ 80% (76)]
    0.1 - 0.2                                           [  0% (0)]
    0.2 - 0.3 #                                         [  3% (3)]
    0.3 - 0.4                                           [  0% (0)]
    0.4 - 0.5                                           [  0% (0)]
    0.5 - 0.6 ##                                        [  5% (5)]
    0.6 - 0.7                                           [  0% (0)]
    0.7 - 0.8                                           [  0% (0)]
    0.8 - 0.9                                           [  0% (0)]
    0.9 - 1.0 ####                                      [ 12% (11)]
    #+end_example
    #+END_EXAMPLE

    Empty buckets print as blank bar + ~[  0% (0)]~ so all 10 rows
    are visible and the user can see "this bucket is empty, not
    missing".

** Sandbox failures

Anything that goes wrong inside the Deno/Pyodide sandbox subprocess:
runaway code that hangs the interpreter, or an outright crash that
loses in-REPL state.

*** DONE Add hard timeout to ~CustomizableRLM._execute_iteration~

   The code interpreter (~PythonInterpreter.execute()~) does
   ~while True: readline()~ with no timeout. When the model generates
   expensive code (e.g. char-by-char JSON parsing of 144k chars in
   Pyodide/WASM), it blocks forever. See [[file:debugging.org][debugging.org]], example 71.

   When this happens the whole RLM call should fail immediately --
   retrying another iteration makes zero sense if the interpreter
   got stuck.

**** Approach

    ~threading.Timer~ kills the Deno subprocess after ~repl_timeout~
    seconds. The ~CodeInterpreterError~ that ~execute()~ raises is
    caught and re-raised as ~REPLTimeoutError~ (a new exception that
    nothing else catches), so it propagates straight out of ~forward()~.

    #+BEGIN_SRC python
    import threading

    class REPLTimeoutError(Exception):
        """Raised when the code interpreter exceeds repl_timeout."""

    # In _execute_iteration, wrap repl.execute:
    try:
        code = _strip_code_fences(action.code)
        timer = threading.Timer(self.repl_timeout, lambda: repl.deno_process.kill())
        timer.start()
        try:
            result = repl.execute(code, variables=dict(input_args))
        finally:
            timer.cancel()
    except (CodeInterpreterError, SyntaxError) as e:
        if not timer.is_alive() and not timer.cancelled():
            # Timer fired -- this is a timeout, not a normal error
            raise REPLTimeoutError(
                f"Code interpreter timed out after {self.repl_timeout}s"
            ) from e
        result = f"[Error] {e}"
    #+END_SRC

    ~REPLTimeoutError~ is not ~CodeInterpreterError~, so nothing in
    RLM or our code catches it -- it bubbles up and fails the call.

    - [X] Add ~repl_timeout~ param to ~CustomizableRLM.__init__~ (default 300),

    - [X] Add ~threading.Timer~ + ~deno_process.kill()~ to ~_execute_iteration~,

*** DONE Catch ~AdapterParseError~ instead of falling back to ~JSONAdapter~
