#+title: DSPy SFT Data Collection - Tasks

* Overview

Collect SFT training data from a DSPy program (teacher LM), filter by
metric, and write JSONL in DSPy's ChatAdapter format -- ready for
finetuning with Unsloth/TRL. Based on the ~SFTDataCollector~ pattern
from [[file:dspy_sft_data_collection.org][dspy_sft_data_collection.org]]. The DSPy module type
(ChainOfThought, RLM, etc.) is configurable via YAML.

** DONE Evaluate RLM predictions via Phoenix traces

   Instead of parsing SFT JSONL output (which contains interleaved RLM
   action steps with unresolvable variable references in SUBMIT calls),
   fetch completed traces from Phoenix and extract final answers directly.

   ~evaluate.py phoenix --config_path=config.yaml --n_traces=10~

   Uses ~traces_endpoint~, ~traces_project~, and ~dataset.path~ from the
   config YAML.

*** Phoenix span schema (verified)

    Tested against local Phoenix at ~http://localhost:6006~.

    - ~from phoenix.client import Client~ / ~Client(base_url=...)~
      (~arize-phoenix-client~ package, already added)
    - ~client.spans.get_spans_dataframe(project_name=..., root_spans_only=True, limit=N)~
    - Root spans have ~name='RLM.forward'~, ~span_kind='CHAIN'~
    - Columns: ~attributes.input.value~, ~attributes.output.value~, ~status_code~, ~start_time~, etc.
    - ~status_code~: ~'OK'~ for completed, ~'ERROR'~ for failed (filter these out)
    - Input is JSON: ~{"input_args": {"context": "...", "query": "..."}}~
      - Does NOT contain dataset row ~id~ -- must match by query text
    - Output is JSON list: ~["Prediction(\n    answer='...VALUE...',\n    trajectory=[...])", [...]]~
      - ~parsed[0]~ is the Prediction repr string
      - Answer extraction: split on ~"',\n    trajectory="~, strip ~"Prediction(\n    answer='"~ prefix
      - Answers may have ~[Answer]\n~ prefix from LongBench-Pro format leak -- strip it
    - Matching to dataset: span's ~input_args.query~ corresponds to
      ~question_nonthinking~ (via prompt template).  Try direct match first;
      if no match, apply ~transform_question~ and retry (handles pre-transform
      spans).  *HACK* -- this is fragile and depends on transform_question
      being idempotent-ish.  Needs a systematic fix: either pass the row ~id~
      through to spans (e.g. as span attribute or in input), or store a
      lookup key in the dataset.
    - Duplicates: multiple spans can match the same dataset row (retries);
      group by matched ~question_nonthinking~, take the latest span per group

*** Approach

    1. Connect to Phoenix: ~Client(base_url=traces_endpoint)~
    2. Fetch root spans: ~client.spans.get_spans_dataframe(project_name=..., root_spans_only=True, limit=N)~
    3. Filter to ~status_code == 'OK'~ only
    4. For each span, extract answer from ~attributes.output.value~ (parse JSON list, split Prediction repr)
    5. Match to dataset row by comparing query text to ~question_nonthinking~ column
    6. Deduplicate: keep most recent span per dataset row (by ~start_time~)
    7. Evaluate using existing LongBench-Pro metric functions

** DONE Export eval results to JSON via ~--output_metrics~

   Add ~--output_metrics=path.json~ flag to ~evaluate~ and ~evaluate_phoenix~.
   Writes a JSON file with two top-level keys:
   - ~summary~: aggregated scores (per-task, per-metric, per-length,
     per-difficulty, overall) -- same info currently printed to stdout

     ~gold_answer~, ~pred_answer~)

*** Pydantic models (in ~config_model.py~)

    Define the JSON output shape as composable Pydantic models.  These
    serve as both validation and serialization -- ~.model_dump()~ gives
    us the JSON directly.

    1. ~GroupStats~ -- reusable stats for any grouping dimension:
       - ~avg: float~
       - ~n: int~

    2. ~ScoredExample~ -- one scored row:
       - ~id: str~
       - ~task: str~ (task code, e.g. "T1.1")
       - ~metric: str~ (metric function name, e.g. "f1_score")
       - ~score: float~
       - ~gold_answer: list[str]~
       - ~pred_answer: list[str]~
       - ~token_length: Optional[str]~ (present if dataset has it)
       - ~difficulty: Optional[str]~ (present if dataset has it)

    3. ~EvalSummary~ -- aggregated scores across dimensions:
       - ~per_task: dict[str, GroupStats]~
       - ~per_metric: dict[str, GroupStats]~
       - ~per_length: Optional[dict[str, GroupStats]]~
       - ~per_difficulty: Optional[dict[str, GroupStats]]~
       - ~overall: GroupStats~
       - ~skipped: int~

    4. ~EvalReport~ -- top-level output written to JSON:
       - ~summary: EvalSummary~
       - ~examples: list[ScoredExample]~

*** Helpers in ~evaluate.py~

    1. ~_score_row(row) -> ScoredExample | None~ -- scores a single
       merged DataFrame row.  Returns ~None~ if no metric or NaN score.
    2. ~_aggregate(results: list[ScoredExample]) -> EvalSummary~ --
       groups scores by task/metric/length/difficulty, builds
       ~GroupStats~ per group.
    3. ~_print_summary(summary: EvalSummary, results, show_errors)~ --
       prints to stdout (same output as today, reads from model fields).
    4. ~_score_predictions~ becomes orchestrator: merge, loop with
       ~_score_row~, call ~_aggregate~ and ~_print_summary~, then if
       ~output_metrics~ is set, build ~EvalReport~ and write
       ~.model_dump()~ as JSON.

** TODO Compare evaluation results across models

   Current eval pipeline has survivorship bias: ~fetch_predictions~ drops
   ERROR traces, and ~_score_predictions~ does an inner join -- so failed
   attempts are invisible.  A model that fails on hard problems but aces
   easy ones looks better than one that attempts everything.

   Create ~compare_evals.py~ that does an outer join across two (or more)
   eval result JSONs to give a fair comparison.

*** Inputs

    - Two or more ~--output_metrics~ JSON files (from ~evaluate.py~)
    - The dataset parquet (for total problem count and metadata)

*** Output

    For each pair of models, report:

    1. *Completion matrix* -- which problems each model scored vs missed:
       - Both scored: N
       - Only model A scored: N (model B failed/not attempted)
       - Only model B scored: N (model A failed/not attempted)
       - Neither scored: N

    2. *Head-to-head on shared problems* (both scored):
       - Model A wins / ties / Model B wins (by score)
       - Average score per model (on the shared subset only)
       - Per-task breakdown

    3. *Adjusted overall scores* -- treat missing predictions as score=0:
       - adjusted_avg = sum(scores) / total_dataset_size
       - This penalizes models that fail to produce answers

    4. *Per-dimension breakdowns* (token_length, difficulty) with
       adjusted scores

*** Approach

    1. Load each metrics JSON: build ~{id: ScoredExample}~ dict per model
    2. Load dataset parquet: get full set of IDs + metadata
    3. Outer join on ID across models
    4. Categorize each ID: scored-by-which-models
    5. Compute head-to-head stats on intersection
    6. Compute adjusted scores using dataset size as denominator
    7. Print comparison table; optionally write to JSON

** TODO Add configurable per-request timeout for LM calls

   When the LLM server drops a connection mid-request (OOM, crash,
   context overflow), the Python process hangs forever in ~CLOSE_WAIT~
   because litellm/httpx has no timeout configured.  Observed with
   finetuned Qwen3.5-4B on llama.cpp -- process hung at example 36
   with a dead socket.

   Add an optional ~timeout~ field to ~LMConfig~ so that each config
   can set a per-request timeout appropriate for the model.  Small
   local models need short timeouts (60s); large API models may need
   longer or no timeout at all.

*** Config changes (~config_model.py~)

    Add to ~LMConfig~:
    - ~timeout: Optional[int] = None~ -- per-request timeout in seconds

*** Code changes (~collect_sft_data.py~)

    In ~main()~, pass ~timeout~ to ~dspy.LM()~ when set:

    #+BEGIN_SRC python
    if cfg.lm.timeout:
        lm_kwargs["timeout"] = cfg.lm.timeout
    #+END_SRC

    litellm accepts ~timeout~ as a kwarg and passes it to the HTTP
    client.  When the timeout fires, litellm raises a timeout error
    which DSPy's ~Evaluate~ counts as a failed example (increments
    error counter) and moves on.
